# TODO (Step by Step)

> 原则：始终保持“可运行 + 可验证”，宁可多拆步骤，不做跨越式开发。

## 1. 批量回测结果增加更完整的 summary
- 目标：让 `run --plan` 的输出不仅有任务列表，还能直接看出这一批里表现最好的组合、最差的组合，以及整体分布特征。
- 方向：在现有 `summary` 基础上继续补充可比较指标，例如收益回撤比、平均收益、收益中位数、最大回撤中位数、胜率等。
- 边界：先只做“汇总与展示”，不改回测引擎的计算逻辑；如果指标定义不统一，先留空字段，等待我确认口径。
- 待确认：是否需要按 `symbol`、`strategy_id`、参数组合分别聚合？

## 2. 参数配置统一化
- 目标：把 `backtest`、`run`、`run_plan` 的参数规则收敛成一套一致的配置模型，减少 CLI 和 JSON 配置之间的分叉。
- 方向：将公共参数抽成统一结构，明确哪些参数来自命令行覆盖、哪些参数来自 `plan` 文件默认值。
- 边界：不强行重写全部命令，只先整理重复逻辑；保留现有命令入口不变。
- 待确认：是否要给 `run_plan` 增加 schema 校验或示例模板？

## 3. 日期窗口与数据切片能力继续扩展
- 目标：让回测不只支持 `start_date` / `end_date`，还可以按更实用的时间切片方式运行。
- 方向：优先考虑“最近 N 天/最近 N 周”、“上市后前 N 根 K 线”等常见窗口，再考虑更复杂的区间组合。
- 边界：先保持日期字段为 `YYYY-MM-DD`，不引入额外时间库，除非确实需要更复杂的时间语义。
- 待确认：窗口优先级是“绝对日期优先”还是“相对窗口优先”？

## 4. 回测加入真实交易约束
- 目标：让回测结果更接近真实交易，而不是只验证策略方向是否正确。
- 方向：逐步加入手续费、滑点、最小交易单位、仓位上限、可用现金约束、单次成交失败处理。
- 边界：先默认不开启或使用保守默认值，避免破坏现有结果的可比性。
- 待确认：手续费和滑点是统一常量，还是按市场/标的分别配置？

## 5. 结果持久化与查询性能优化
- 目标：避免结果文件数量增长后，leaderboard 和历史查询越来越慢。
- 方向：增加一个轻量索引层，或者把结果写入更适合查询的存储；同时保留当前 JSON 作为可读备份。
- 边界：先不替换现有文件格式，确保历史结果还能直接读取。
- 待确认：是否接受引入 SQLite 之类的本地数据库，还是继续坚持纯文件方案？

## 6. 增加面向行为的测试
- 目标：把“能编译”提升到“关键行为不会回退”。
- 方向：补充针对日期过滤、任务去重、非法日期、summary 生成、clean 命令边界的测试。
- 边界：优先测规则和输出，不要把测试写成对内部实现细节的强绑定。
- 待确认：测试目标是只覆盖 CLI 入口，还是把回测引擎的核心行为也一起纳入？

## 7. CLI 体验继续收敛
- 目标：让命令更好记、更少翻文档。
- 方向：补充子命令帮助、参数默认值提示、常见组合示例，以及 run 完成后的简短摘要输出。
- 边界：保留当前命令兼容性，不做破坏性重命名；如果要改命令结构，先给出过渡期。
- 待确认：是否需要给 `help` 做成按主题分组的输出？
